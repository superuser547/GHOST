# REQUIREMENTS

## Введение

### Назначение документа

Документ фиксирует единый набор **функциональных требований (ФТ)** и **технического задания (ТЗ)** для проекта GHOST — локального веб‑приложения для сборки и генерации академических отчётов по требованиям ГОСТ и НИТУ МИСИС.

### Область применения

Документ предназначен для:

- разработки и сопровождения проекта;
- проверки соответствия реализации требованиям;
- согласования поведения интерфейса и формируемого документа.

Система работает **офлайн**, не использует внешние сервисы и доступна в кроссплатформенной среде (Windows/macOS/Linux, при необходимости через Docker).

## 1. Назначение системы и сценарии использования (ФТ)

### 1.1. Краткое описание системы

Система — локальное веб‑приложение, в котором пользователь:

- собирает структуру отчёта из блоков по правилам МИСИС/ГОСТ;
- заполняет содержание блоков;
- видит ошибки и предупреждения по правилам оформления;
- экспортирует результат в **DOCX**.

Экспорт в PDF рассматривается как **будущая возможность** и в первой версии может отсутствовать.

### 1.2. Роли

Единственная роль: **Пользователь** (студент/преподаватель/автор отчёта).

### 1.3. Основные сценарии

#### 1.3.1. Создание нового отчёта

1. Пользователь открывает приложение.
2. На экране отображаются:
   - панель параметров (метаданные и титульный лист);
   - панель блоков (палитра);
   - конструктор отчёта;
   - панель ошибок и предупреждений.
3. Пользователь заполняет метаданные и создаёт структуру отчёта.
4. Система валидирует структуру и показывает ошибки/предупреждения.
5. Пользователь исправляет замечания и выполняет экспорт в DOCX.

#### 1.3.2. Настройка титульного листа

Пользователь указывает:

- тип работы (ПР/ЛР/реферат/курсовая и т. д.);
- дисциплину;
- тему;
- ФИО студента, группу, семестр;
- направление подготовки (код и название);
- кафедру;
- ФИО преподавателя;
- дату.

Значения сохраняются в `localStorage` и используются при формировании титульного листа.

#### 1.3.2.1. Пользовательский шаблон титульного листа

- Пользователь может загрузить **одностраничный DOCX‑файл** с собственным оформлением титульного листа (образец по методичке МИСИС).
- В этом файле он размещает **плейсхолдеры** для автоматической подстановки данных, например:
  - `{{STUDENT_NAME}}` — ФИО студента;
  - `{{GROUP}}` — номер группы;
  - `{{SEMESTER}}` — семестр;
  - `{{DIRECTION_CODE}}` — код направления подготовки;
  - `{{DIRECTION_NAME}}` — название направления подготовки;
  - `{{DEPARTMENT}}` — название кафедры;
  - `{{DISCIPLINE}}` — дисциплина;
  - `{{WORK_TYPE}}` — тип работы (Практическая работа / Лабораторная работа / Реферат / собственное название);
  - `{{WORK_NUMBER}}` — номер работы (если применимо);
  - `{{TOPIC}}` — тема работы;
  - `{{TEACHER_NAME}}` — ФИО преподавателя;
  - при необходимости могут быть добавлены дополнительные плейсхолдеры (год, город и т. п.).
- Путь к файлу шаблона титульного листа и введённые пользователем значения полей сохраняются в `localStorage` и восстанавливаются при повторном открытии приложения.

#### 1.3.3. Сборка структуры отчёта

Пользователь добавляет блоки из панели, меняет порядок, редактирует содержимое и управляет вложенностью (разделы/подразделы/приложения).

#### 1.3.4. Просмотр и исправление ошибок

Система отображает ошибки (блокируют экспорт) и предупреждения (не блокируют экспорт). При клике на сообщение фокусируется соответствующий блок.

#### 1.3.5. Экспорт в DOCX

1. Пользователь нажимает кнопку «Экспорт в DOCX».
2. Система валидирует отчёт и при отсутствии ошибок формирует `.docx` через **python-docx**.
3. Система возвращает файл для скачивания.

После завершения экспорта интерфейс показывает подсказку:

> «Откройте файл в Word и выполните: (1) сформируйте/обновите оглавление средствами Word по стилям заголовков; (2) включите нумерацию страниц снизу по центру, начиная с введения, без номера на титульном листе и содержании. Все остальные требования оформления соблюдены автоматически.»

#### 1.3.6. Сохранение и загрузка проекта

Пользователь может сохранить отчёт в JSON‑файл и загрузить его позже. В первой версии это возможно на фронтенде (экспорт/импорт JSON) без серверного хранения.

## 2. Модель данных и типы блоков

### 2.1. Общая модель

Отчёт описывается сущностью **Report**, которая содержит метаданные и список блоков:

- `meta` — параметры отчёта и данные титульного листа;
- `blocks` — список блоков с вложенной структурой.

Пример Pydantic‑схем (фрагмент):

```python
from pydantic import BaseModel
from typing import List, Optional, Literal, Union

class ReportMeta(BaseModel):
    preset: Literal["MISIS"]

    # Тип работы:
    # - "practice"      — практическая работа
    # - "lab"           — лабораторная работа
    # - "essay"         — реферат
    # - "custom"        — пользовательский тип (см. поле work_type_custom)
    work_type: Literal["practice", "lab", "essay", "custom"]

    # Если work_type == "custom", здесь хранится текстовое название работы,
    # которое будет подставлено в титульный лист.
    work_type_custom: Optional[str] = None
    discipline: str
    topic: str
    student_name: str
    student_group: str
    semester: str
    direction_code: str
    direction_name: str
    department: str
    teacher_name: str
    submission_date: str

class SectionBlock(BaseModel):
    type: Literal["SECTION"]
    title: str
    special_kind: Optional[Literal["INTRO", "CONCLUSION", "REFERENCES"]]
    children: List["ReportBlock"]

class TextBlock(BaseModel):
    type: Literal["TEXT"]
    text: str

ReportBlock = Union[SectionBlock, TextBlock, ...]

class Report(BaseModel):
    meta: ReportMeta
    blocks: List[ReportBlock]
```

### 2.2. Типы блоков

Поддерживаемые блоки:

1. **Титульный лист** (`TITLE_PAGE`)
2. **Раздел** (`SECTION`)
3. **Подраздел** (`SUBSECTION`) уровней 2–3
4. **Текст** (`TEXT`)
5. **Список** (`LIST`)
6. **Таблица** (`TABLE`)
7. **Рисунок** (`FIGURE`)
8. **Список источников** (`REFERENCES`)
9. **Приложение** (`APPENDIX`)

Отдельного блока формул нет — формулы добавляются пользователем вручную в Word.

## 3. Требования к интерфейсу (Frontend)

### 3.1. Панель параметров

- выбор пресета оформления (по умолчанию MISIS);
- поля метаданных (тип работы, дисциплина, тема);
  - Тип работы:
    - выпадающий список со значениями:
      - «Практическая работа»;
      - «Лабораторная работа»;
      - «Реферат»;
      - «Другое (своё название)»;
    - при выборе «Другое (своё название)» отображается дополнительное текстовое поле для ввода произвольного названия (например, «Курсовая работа»);
    - введённое значение сохраняется в `ReportMeta.work_type_custom` и подставляется в титульный лист;
    - для предопределённых типов название формируется стандартно (например, «Практическая работа №N»).
- поля титульного листа (ФИО, группа, направление, кафедра, преподаватель и т. д.);
- сохранение значений в `localStorage` и восстановление при следующем запуске.

### 3.2. Панель блоков (палитра)

- отображает доступные типы блоков;
- поддерживает добавление кликом и drag&drop.

### 3.3. Конструктор отчёта

- визуальное дерево блоков с вложенностью;
- редактирование содержимого каждого блока;
- перемещение и удаление блоков.

### 3.4. Панель ошибок и предупреждений

- список ошибок (блокируют экспорт) и предупреждений;
- переход к соответствующему блоку по клику.

### 3.5. Экспорт

- кнопка «Экспорт в DOCX»;
- после успешного экспорта отображается подсказка о ручных действиях в Word:
  - сформировать/обновить оглавление;
  - включить нумерацию страниц снизу по центру, начиная с введения, без номера на титульном листе и содержании.

Экспорт в PDF — необязательная функция и может отсутствовать в первой версии.

## 4. Требования к бэкенду и архитектуре (ТЗ)

### 4.1. Технологии

- язык: **Python 3.11+**;
- фреймворк: **FastAPI**;
- генерация документов: **python-docx** (при необходимости с использованием low‑level XML).

### 4.2. Генерация DOCX

- `.docx` формируется **полностью через python-docx**;
- структура документа собирается из модели `Report` и блока(ов);
- параметры оформления (шрифты, поля, интервалы, стили заголовков, подписи таблиц и рисунков) задаются в коде и/или конфигурации репозитория;
- допускаются технические шаблоны в `assets/templates/`, если они используются как базовый каркас, но требования к стилям и форматированию остаются документированными и контролируемыми.

**Оглавление и нумерация страниц в первой версии не формируются автоматически.** Пользователь выполняет эти действия вручную в Word.

### 4.3. Основные модули

- модели (`Report`, блоки, метаданные);
- валидация;
- сервис генерации DOCX (python-docx);
- сервис сохранения/загрузки проектов (если реализован на бэкенде).

### 4.4. Эндпоинты (минимально необходимые)

- `GET /api/v1/presets` — список пресетов оформления;
- `POST /api/v1/reports/validate` — валидация отчёта;
- `POST /api/v1/reports/export/docx` — экспорт в DOCX;
- `POST /api/v1/reports/export/pdf` — **опционально**;
- эндпоинты сохранения/загрузки проектов — **опционально**.

## 5. Требования к формированию DOCX по МИСИС

Раздел фиксирует **обязательные числовые параметры и правила оформления**, которые должны соблюдаться при генерации `.docx` средствами python-docx.

Каждое правило имеет статус:

- **Обязательное при генерации** — документ формируется строго по параметрам;
- **Проверки/подсветка** — правило проверяется валидатором;
- **Рекомендации** — правило описано в методичке, но автоматический контроль ограничен; система может показывать предупреждение.

### 5.1. Параметры страницы и ориентация

**Обязательное при генерации**

- Формат: **A4**.
- Поля (книжная ориентация):
  - левое: **3,0 см**;
  - правое: **1,5 см**;
  - верхнее: **2,0 см**;
  - нижнее: **2,0 см**.
- Поля (альбомная ориентация для широких таблиц/рисунков):
  - левое: **2,0 см**;
  - правое: **2,0 см**;
  - верхнее: **3,0 см**;
  - нижнее: **1,5 см**.
- Альбомная ориентация применяется только при необходимости и с теми же требованиями к форматированию текста, что и для книжной.

**Проверки/подсветка**

- контроль корректности полей и обоснованности альбомной ориентации.

### 5.2. Нумерация страниц и правило «≤ 25% пустого места»

**Нормативные требования**

- Нумерация страниц начинается с первой страницы отчёта.
- Номер не отображается на титульном листе и странице содержания.
- Номер отображается **снизу по центру**.

**Важно:** в первой версии приложения нумерация страниц **не формируется автоматически**. Пользователь включает её вручную в Word.

**Проверки/подсветка**

- попытка оценить превышение 25% пустого места на странице (warning).

### 5.3. Основной текст отчёта

**Обязательное при генерации**

- Шрифт: **Times New Roman, 12 pt**.
- Выравнивание: **по ширине**.
- Отступы абзаца:
  - слева: **0 см**;
  - справа: **0 см**;
  - первая строка: **1,25 см**.
- Межстрочный интервал: **1,5**.
- Интервалы до и после абзаца: **0 pt** (все вертикальные интервалы задаются только за счёт настроек заголовков, таблиц, рисунков и других специальных элементов).
- Не допускается:
  - использование «пустых» абзацев для создания вертикальных отступов;
  - использование других шрифтов и кеглей в основном тексте (кроме явно оговорённых случаев, например подписи, таблицы).

**Проверки/подсветка**

- при наличии подряд идущих пустых абзацев система может выдавать предупреждение (warning) как рекомендацию для пользователя.

### 5.4. Заголовки и структура разделов

**Обязательное при генерации**

- Заголовки уровней 1–3: **Times New Roman, 12 pt**, **полужирный**, без подчёркивания, межстрочный интервал **1,5**.
- Для всех заголовков уровней 1–3 абзацный отступ (первая строка) равен **0 см**.
- Заголовки структурных разделов печатаются **прописными**:
  - «СОДЕРЖАНИЕ», «ВВЕДЕНИЕ», «ЗАКЛЮЧЕНИЕ», «СПИСОК ИСПОЛЬЗОВАННЫХ ИСТОЧНИКОВ», «ПРИЛОЖЕНИЕ».
- Заголовки ключевых структурных разделов:
  - **«ВВЕДЕНИЕ»**,
  - **«ЗАКЛЮЧЕНИЕ»**,
  - **«СПИСОК ИСПОЛЬЗОВАННЫХ ИСТОЧНИКОВ»**,
  - **«ПРИЛОЖЕНИЕ»** (без номерного префикса вида «1»)
  оформляются **без номерного префикса** и **выравниваются по центру**.
- Остальные заголовки разделов первого уровня:
  - нумеруются по схеме многоуровневой нумерации (1, 2, 3 …);
  - выравниваются **по левому краю**.
- Многоуровневая нумерация задаётся в стилях заголовков:
  - уровень 1: `1`, `2`, `3` …;
  - уровень 2: `1.1`, `1.2` …;
  - уровень 3: `1.1.1` …;
  - позиция номера: **1,25 см**;
  - отступ текста после номера: **1,25 см**;
  - после номера — пробел.
- Пустые абзацы между заголовком и текстом/подзаголовком не допускаются.

**Интервалы вокруг заголовков**

| Сценарий | Интервал до | Интервал после |
| --- | --- | --- |
| Заголовок в начале страницы → сразу текст | 0 pt | 24 pt |
| Заголовок в начале страницы → сразу подзаголовок | 0 pt | 0 pt |
| Заголовок среди текста → сразу подзаголовок | 24 pt | 0 pt |
| Заголовок среди текста → сразу текст | 24 pt | 24 pt |

**Проверки/подсветка**

- соблюдение сценариев интервалов и отсутствие пустых абзацев;
- корректность схемы нумерации и привязки к стилям заголовков.

### 5.5. Списки

**Обязательное при генерации**

- Абзац списков:
  - шрифт **Times New Roman, 12 pt**;
  - выравнивание по ширине;
  - отступы слева/справа **0 см**;
  - первая строка **0 см**;
  - межстрочный интервал **1,5**;
  - интервалы до/после **0 pt**.
- Маркированный список:
  - маркер **«–»** (длинное тире);
  - позиция маркера **0 см**;
  - отступ текста **1,25 см**;
  - после маркера пробел.
- Нумерованный список:
  - формат **`1)`**;
  - позиция номера **0 см**;
  - отступ текста **1,25 см**;
  - после номера табуляция.
- Вложенные перечисления:
  - основной формат — **русские буквы** с закрывающей скобкой (`а)`, `б)`), без букв **ё, э, й, о, ч, ъ, ы, ь**;
  - допускается формат `1)` на вложенном уровне при необходимости.

**Проверки/подсветка**

- неверные маркеры/нумерация → ошибка;
- некорректная схема вложенных списков → предупреждение или ошибка.

Примечание. В презентациях МИСИС используется формулировка о том, что элементы списка не должны иметь «отступов и выступов». В данном контексте имеется в виду запрет на ручные отступы (пробелами/табуляцией). Фактические отступы задаются стилем списка (позиция маркера 0 см, отступ текста 1,25 см).

### 5.6. Таблицы

**Обязательное при генерации**

- Таблица выравнивается по центру страницы.
- Таблица создаётся с шириной 100% области текста (автоподбор по ширине окна документа).
- Шрифт содержимого таблиц: **Times New Roman, 10 pt**, интервал **1,0**.
- Текст в ячейках по умолчанию выравнивается по левому краю; для заголовков допускается центр.
- Подпись таблицы:
  - формат **`Таблица X – Название`**;
  - шрифт **Times New Roman, 12 pt**;
  - выравнивание по левому краю;
  - интервалы до/после **0 pt**, интервал **1,5**.
- После таблицы: интервал перед первым абзацем **6 pt**.
- Перед и после таблицы не вставляются пустые абзацы; вертикальный интервал задаётся только стилем абзаца после таблицы (интервал перед первым абзацем после таблицы — 6 pt).
- Запрещено:
  - колонка «№ п/п»;
  - вставка таблицы как изображения.
  - использование заливки ячеек (цветного фона).
- Таблица обязана иметь шапку.
- При переносе на следующую страницу:
  - строка «Продолжение таблицы X»;
  - повтор заголовка.

**Проверки/подсветка**

- отсутствие шапки, колонка «№ п/п», таблица как изображение → ошибка;
- отсутствие строки «Продолжение таблицы X» → предупреждение.

### 5.7. Рисунки

**Обязательное при генерации**

- Рисунок выравнивается по центру страницы.
- Рисунок размещается в пределах полей страницы и не может выходить за их границы.
- Подпись:
  - формат **`Рисунок X – Название`**;
  - **Times New Roman, 12 pt**, интервал **1,5**;
  - выравнивание по центру;
  - интервалы до/после **0 pt**.
- Перед и после рисунка не вставляются пустые абзацы; отступы задаются стилями абзацев до и после рисунка.
- Рисунок должен располагаться непосредственно после первого упоминания в тексте.

**Проверки/подсветка**

- отсутствие подписи или неправильный формат → ошибка;
- раздел, заканчивающийся рисунком без поясняющего текста → ошибка.

**Рекомендации**

- рекомендуется оформлять ссылки на рисунки как перекрёстные ссылки средствами Word (вставка → перекрёстная ссылка) с последующим обновлением F9; автоматическая генерация перекрёстных ссылок приложением не выполняется.

### 5.8. Список использованных источников и ссылки

**Обязательное при генерации**

- Ссылки в тексте в формате **`[1]`, `[2]`…**.
- Каждая ссылка соответствует записи в списке источников.
- Оформление списка источников:
  - **Times New Roman, 12 pt**;
  - выравнивание по ширине;
  - отступ слева/справа: **0 см**;
  - первая строка: **0 см**;
  - межстрочный интервал **1,5**;
  - интервалы до/после **0 pt**.

**Проверки/подсветка**

- ссылка без записи в списке → ошибка.

**Рекомендации**

- источники старше 5 лет помечаются предупреждением.
- при необходимости допускается использование висячего отступа для строк, следующих за первой, средствами Word, но приложение это автоматически не настраивает.

### 5.9. Приложения

**Обязательное при генерации**

- Обозначение приложений: **А, Б, В…**, без **Ё, З, Й, О, Ч, Ъ, Ы, Ь**.
- Заголовок приложения:
  - формат **«ПРИЛОЖЕНИЕ А»** (прописные, полужирный, по центру);
  - при наличии названия: **«ПРИЛОЖЕНИЕ А – Название»**.
- Нумерация в приложениях:
  - рисунки: **«Рисунок А.1 – …»**;
  - таблицы: **«Таблица А.1 – …»**.

**Проверки/подсветка**

- неверная буква или дубликат → ошибка.

### 5.10. Содержание (оглавление)

**Нормативные требования к виду оглавления**

- Заголовок «СОДЕРЖАНИЕ»:
  - прописными, полужирный;
  - **Times New Roman, 12 pt**, по центру;
  - интервал после **24 pt**;
  - межстрочный интервал **1,5**.
- Стили строк оглавления (TOC 1 / TOC 2 / TOC 3):
  - TOC 1:
    - **Times New Roman, 12 pt**, межстрочный интервал **1,5**, интервалы до/после **0 pt**;
    - отступ слева: **0 см**; отступ справа: **0 см**;
    - первая строка: **0 см**.
  - TOC 2:
    - отступ слева: **0,62 см**; отступ справа: **0 см**;
    - первая строка: **0 см**.
  - TOC 3:
    - отступ слева: **1,25 см**; отступ справа: **0 см**;
    - первая строка: **0 см**.
  - TOC n+1: отступ слева = отступ уровня n + **0,625 см**.

**Важно:** в первой версии приложения оглавление **не формируется автоматически**. Пользователь создаёт/обновляет его вручную в Word по стилям заголовков.

### 5.11. Ручные действия пользователя в Word

После генерации `.docx` пользователь должен:

1. Открыть документ в Word.
2. Сформировать/обновить оглавление средствами Word (по стилям заголовков).
3. Включить нумерацию страниц снизу по центру, начиная с введения, без номера на титульном листе и содержании.
4. Рекомендуется переименовать файл в соответствии с методическими указаниями МИСИС, например: Фамилия_группа_аббревиатура дисциплины_тип работы №номер (пример: Ivanov_BBI-24-3_OTS_PR№1). Приложение может подсказать рекомендуемое имя, но не обязано применять его автоматически.

Эти шаги должны быть отражены в UI подсказке после экспорта.

### 5.12. Титульный лист и пользовательский шаблон

**Обязательное при генерации**

- Титульный лист формируется на основе **пользовательского DOCX‑шаблона**, который содержит ровно одну страницу.
- Бэкенд:
  - открывает загруженный шаблон титульного листа через `python-docx`;
  - находит в нём плейсхолдеры по заранее определённым маркерам (см. раздел 1.3.2.1);
  - заменяет плейсхолдеры на значения из `ReportMeta` без изменения форматирования абзацев, стилей и расположения элементов;
  - полученную страницу титульного листа **добавляет в начало** итогового документа перед основным текстом отчёта.
- Форматирование титульного листа (шрифты, интервалы, выравнивание) полностью определяется пользовательским шаблоном; генератор отвечает только за корректную подстановку значений и сохранение структуры.
- Поля, заполняемые преподавателем (оценка, дата проверки, подпись и подобные элементы), в шаблоне не заполняются автоматически и остаются пустыми в сгенерированном титульном листе.

**Проверки/подсветка**

- при отсутствии одного или нескольких плейсхолдеров в шаблоне титульного листа система может:
  - либо выдавать предупреждение (warning) с перечислением отсутствующих плейсхолдеров;
  - либо использовать только те плейсхолдеры, которые найдены, без блокировки экспорта.

### 5.13. Формулы (справочно)

Формулы:

- вставляются только как формулы Word, не как рисунки;
- размещаются по центру строки;
- нумеруются арабскими цифрами в круглых скобках справа;
- под формулой даются пояснения, начиная со слова «где» (с маленькой буквы).

Эти требования относятся к ручному оформлению и не контролируются автоматически.

## 6. Правила валидации (ошибки и предупреждения)

### 6.1. Ошибки (блокируют экспорт)

- `REQUIRED_SECTIONS_PRESENT` — отсутствуют обязательные структурные элементы: титульный лист, содержание (страница «СОДЕРЖАНИЕ»), введение, основная часть (разделы 1, 2, 3…), заключение, список использованных источников. Проверка наличия страницы «СОДЕРЖАНИЕ» может быть реализована как отдельное правило (ошибка или предупреждение).
- `SECTION_ORDER` — нарушен порядок ключевых разделов.
- `SECTION_ENDS_WITH_MEDIA` — раздел заканчивается рисунком/таблицей/списком без завершающего текста.
- `NON_EMPTY_LISTS` — список пустой.
- `FIGURE_HAS_CAPTION` — у рисунка нет подписи.
- `TABLE_HAS_CAPTION` — у таблицы нет подписи.
- `APPENDIX_LABELS_UNIQUE` — повторяющиеся обозначения приложений.
- `FIGURE_TABLE_NUMBERING_CONSISTENT` — дублирование номеров таблиц/рисунков.
- `REFERENCES_PRESENT_IF_NEEDED` — требуется список источников, но блока нет.
- `LIST_MARKER_FORMAT` — неверный формат маркера или нумерации списка.
- `TABLE_REQUIREMENTS` — таблица без шапки или с запрещённой колонкой.
- `FIGURE_REQUIREMENTS` — подпись рисунка не соответствует формату.
- `REFERENCES_LINKS_VALID` — ссылка в тексте без записи в списке.
- `HEADING_SPACING_SCENARIOS` — нарушены интервалы вокруг заголовков.
- `HEADING_NUMBERING_STYLE` — некорректная схема нумерации заголовков.

### 6.2. Предупреждения (не блокируют экспорт)

- `APPENDIX_LABELS_ORDER` — приложения расположены не по алфавиту.
- `LIST_OF_REFERENCES_NOT_EMPTY` — список источников пустой.
- `REFERENCES_AGE_WARNING` — источник старше 5 лет.
- `REPORT_EMPTY_SPACE_LIMIT` — вероятное превышение 25% пустого места.
- `WORK_GOAL_RECOMMENDED` — в начале отчёта не обнаружен явный формулируемый раздел/абзац с «Целью работы» (рекомендуется по методическим указаниям).

## 7. Нефункциональные требования

### 7.1. Портируемость

- Windows 10+ / macOS / Linux.
- Возможность локального запуска и через Docker.

### 7.2. Производительность

- Типичный объём отчёта: 50–100 страниц, десятки таблиц/рисунков.
- Валидация и UI‑обновления — в пределах 200–300 мс на типичном отчёте.
- Экспорт в DOCX — секунды, а не минуты.

### 7.3. Безопасность

- Нет сетевых вызовов и внешних API.
- Файлы и изображения обрабатываются локально.

### 7.4. Качество кода

- типизация Python;
- форматирование (black), линтинг (ruff);
- типизированный фронтенд на TypeScript, линтеры и форматтеры.

## 8. Ограничения и то, что не реализуется

- оформление формул (вставка, форматирование, нумерация, пояснения) полностью остаётся на стороне пользователя и выполняется в Word в соответствии с методическими указаниями МИСИС. Приложение не создаёт, не редактирует и не валидирует формулы;
- система не выполняет проверку на антиплагиат и не формирует отчёты об антиплагиате. Если дисциплина требует приложить отчёт об антиплагиате (с QR‑кодом), его получение и прикрепление к работе выполняется пользователем вручную (например, как отдельное приложение);
- нет AI/LLM‑функций;
- нет автоматического построения оглавления и нумерации страниц в текущей версии (только вручную в Word);
- не используются коммерческие библиотеки (Aspose, Spire и т. п.).
